# Chrome 侧边栏 + Conductor：我们需要什么

## 我们正在构建的内容

现在当 Claude 在 Conductor 工作区中运行时 —— 编辑文件、测试、浏览你的应用 —— 你只能从 Conductor 的聊天窗口观看。如果 Claude 正在对你的网站进行 QA，你会看到工具调用滚动，但无法真正*看到浏览器*。

我们构建了一个 Chrome 侧边栏来修复这个问题。当你运行 `$B connect` 时，Chrome 会打开一个侧面板，实时显示 Claude 正在做的一切。你可以在侧边栏中输入消息，Claude 会对其作出响应 —— "点击注册按钮"，"去设置页"，"总结你看到的"。

问题是：侧边栏目前运行自己独立的 Claude 实例。它看不到主 Conductor 会话在做什么，主会话也看不到侧边栏在做什么。它们是两个不相互通信的独立代理。

修复很简单：让侧边栏成为 Conductor 会话的一个*窗口*，而不是一个独立的东西。

## 我们需要 Conductor 提供的 3 件事

### 1. 让我们看到代理在做什么

我们需要一种订阅活跃会话事件的方式。类似于 SSE 流或 WebSocket 的东西，当事件发生时向我们发送事件：

- "Claude 正在编辑 `src/App.tsx`"
- "Claude 正在运行 `npm test`"
- "Claude 说：我会修复 CSS 问题..."

侧边栏已经知道如何渲染这些事件 —— 工具调用显示为紧凑徽章，文本显示为聊天气球。我们只需要一根从 Conductor 会话到我们的扩展的管道。

### 2. 让我们向会话发送消息

当用户在 Chrome 侧边栏中输入 "点击另一个按钮" 时，该消息应该出现在 Conductor 会话中，就像用户在工作区聊天中输入了一样。代理在下一轮接收并对其采取行动。

这是魔法时刻：用户正在观看 Chrome，发现问题，在侧边栏中输入纠正，Claude 响应 —— 用户无需切换窗口。

### 3. 让我们从目录创建工作区

当 `$B connect` 启动时，它会创建一个用于文件隔离的 git worktree。我们希望将该 worktree 注册为 Conductor 工作区，以便用户可以在 Conductor 的文件树中看到侧边栏代理的文件更改。这也为多个浏览器会话奠定了基础，每个都有各自的工作区。

## 为什么这很重要

今天，`/qa` 和 `/design-review` 感觉像一个黑盒。Claude 说 "我发现了 3 个问题"，但你看不到它在看什么。通过连接到 Conductor 的侧边栏：

- **你实时观看 Claude 测试你的应用** —— 每次点击、每次截屏都出现在 Chrome 中，你可以随时观看
- **你可以打断** —— "不，测试手机视图" 或 "跳过那个页面" —— 无需切换窗口
- **一个代理，两种视图** —— 编辑你的代码的同一个 Claude 也在控制浏览器。无上下文复制，无状态陈旧

## 已构建的内容（gstack 侧）

我们这边一切都已完成并交付：

- 运行 `$B connect` 时自动加载的 Chrome 扩展
- 自动打开的侧面板（用户零设置）
- 流式事件渲染器（工具调用、文本、结果）
- 带消息队列的聊天输入
- 带状态横幅的重连逻辑
- 带持久聊天历史记录的会话管理
- 代理生命周期（生成、停止、终止、超时检测）

我们这边唯一的改变：将数据源从 "本地 `claude -p` 子进程" 换成 "Conductor 会话流"。扩展代码保持不变。

**估计工作量：** Conductor 工程 2-3 天，gstack 集成 1 天。
