## 问题

Louise 阅读 gstack 评审输出的疲劳感来自两个来源：

1. **术语密度** —— 技术术语出现时没有解释。*已在 V1 中解决（ELI10 写作）。*
2. **打断量** —— `/autoplan` 运行 4 个阶段（CEO + Design + Eng + DX），每个阶段有 5–10 个 AskUserQuestion 提示。总计约 30–50 个提示，历时 ~45 分钟。非技术用户在约 10–15 次打断时就会放弃。**这是 V1.1。**

翻译本身不能修复打断量。一个被打断的翻译仍然是打断。修复需要改变发现项浮现的时机，而不仅仅是措辞方式。

## 为何被提取（V1 第三次工程审查 + Codex 第二轮审查揭示的结构性缺口）

在 V1 规划期间，起草了一个节奏工作流：对发现项排序、自动接受双向门（two-way doors）、每个审查阶段最多 3 个 AskUserQuestion 提示、自动接受项的"静默决策"块、"flip <id>"命令用于事后重新打开自动接受的决策。第三次工程审查 + 第二轮 Codex 审查揭示了 10 个无法通过计划文本编辑修复的缺口：

1. **会话状态模型未定义。** 节奏需要每个阶段的状态（哪些发现项浮现了、哪些自动接受了、哪些用户可以翻转）。V1 有每次技能调用时的术语注释状态，但没有用于每阶段节奏记忆的存储后端。
2. **问题日志中缺少阶段标识符。** 静默的工程审查 #8 曾希望在 >3 个提示出现在同一阶段时发出警告。V0 的 `question-log.jsonl` 没有 `phase` 字段。V1 声称"无模式变更"——与执行目标矛盾。
3. **问题注册表 ≠ 发现项注册表。** V0 的 `scripts/question-registry.ts` 涵盖*问题*（在技能定义时注册）。评审发现项是*动态的*（在运行时发现）。通过注册表执行 `door_type: one-way` 不涵盖临时发现项。对于代理在评审中途生成的发现项，单向门安全性不可执行。
4. **作为散文的节奏无法反转现有控制流。** V1 计划在序言抽象中添加"排序发现项，然后提问"的规则。但现有技能模板如 `plan-eng-review/SKILL.md.tmpl` 有每段的 STOP/AskUserQuestion 序列。序言中的散文规则无法可靠覆盖硬编码的每段 STOP。行为改变是顺序，而非提示措辞。
5. **翻转机制没有实现。** "回复 `flip <id>` 以更改"只是散文。没有命令解析器、没有状态存储、没有重放行为。如果对话压缩且静默决策块离开上下文，原始决策就丢失了。
6. **迁移提示本身就是一个打断。** V1 的升级后迁移提示（提供恢复 V0 散文）计入 V1.1 试图减少的打断预算。V1.1 必须决定：从预算中豁免，还是计为第 N 个打断之一？
7. **首次运行序言提示也计入。** Lake 介绍、遥测、主动、路由注入 —— Louise 在首次运行时全部看到它们。它们是第一个真正技能运行之前的打断。V1.1 必须审计哪些对新用户是关键的，哪些可以推迟到第 N 个会话。
8. **排序公式未根据真实数据校准。** V1 考虑了 `product 0-8`（有缺陷：`{0,1,2,4,8}` 分布），然后是 `sum 0-6` 且阈值 ≥ 4。但两者都未根据实际发现项分布验证。V1.1 应给 V0 问题日志打桩，测量真实发现项的样子，然后校准。
9. **"每个单向门都浮现"与"每阶段最多 3 个"矛盾。** 单向门上限 = 无限（安全）；双向门上限定为 3。但计划同时有两条规则而没有明确的优先级。V1.1 必须声明：无论阶段预算如何，单向门都无限期浮现。
10. **未定义的验证值。** V1 计划有"静默决策块 ≥ N 个条目"，其中 N 从未定义，以及吞吐量 JSON 中 `active: true` 字段从未定义。V1.1 给出具体值。

## V1.1 范围

1. **定义会话状态模型。** 每次技能调用 vs 每阶段 vs 每次对话。存储后端：可能是一个 JSON 文件，位于 `~/.gstack/sessions/<session_id>/pacing-state.json`，记录每个阶段哪些发现项浮现 vs 自动接受。清理：与序言中现有会话跟踪相同的 TTL。
2. **向 question-log.jsonl 模式添加 `phase` 字段。** 将每个 AskUserQuestion 按来源审查阶段分类（CEO / Design / Eng / DX / 其他）。迁移：现有条目默认为 `"unknown"`。非破坏性模式扩展。
3. **扩展动态发现项的注册表覆盖。** 两个选项，在 CEO 审查期间选择：
   - (a) 加宽 `scripts/question-registry.ts` 以允许运行时注册（临时 ID 仍然被记录 + 分类）。
   - (b) 添加辅助运行时分类器 `scripts/finding-classifier.ts`，使用模式匹配将发现项文本映射到风险层级。
4. **将节奏从序言散文移入技能模板控制流。** 更新每个审查技能模板以：(i) 内部完成阶段，(ii) 通过 `gstack-pacing-rank` 二进制对发现项排序，(iii) 发出最多 3 个 AskUserQuestion 提示，(iv) 发出静默决策块包含剩余部分。不是序言规则 —— 每个模板中的显式序列。
5. **翻转机制实现。** 新的二进制 `bin/gstack-flip-decision`。命令解析器接受用户消息中的 `flip <id>`。在 pacing-state.json 中查找原始决策。作为显式 AskUserQuestion 重新打开。新选择持久存在。
6. **迁移提示预算决定。** 显式规则：一次性迁移提示免受每阶段打断预算限制。理由：它们在审查阶段开始前触发，而非期间。
7. **首次运行序言审计。** 审计 lake 介绍、遥测、主动、路由注入。对每一项：对新用户是否关键，或可推迟？可能结果：将除 lake 介绍外的全部抑制，直到第 2+ 个会话。通过 `/plan-tune first-run` 命令提供剩余选项供用户自愿调用。
8. **排序阈值校准。** 为 V0 的问题日志打桩（已在运行，有历史）。测量近期 CEO + Eng + DX + DX 评审中 `severity × irreversibility × user-decision-matters` 的实际分布。基于真实数据选择阈值。目标：~20% 发现项浮现，~80% 自动接受。
9. **显式规则：单向门无限。** 硬编码在技能模板散文中："无论阶段打断预算如何，单向门都浮现。"双向发现项每阶段上限定为 3。
10. **具体验证值。** 为静默决策定义 `N`（例如，非平凡计划预期 ≥ 5 个条目），通过具体字段名定义吞吐量 JSON 模式。

## V1.1 验收标准

- **打断次数：** Louise（或类似的非技术协作者）在可比于 V0 基线的计划上端到端重跑 `/autoplan`。AskUserQuestion 次数 ≤ V0 基线的 50%。（V1 捕获此基线转录供 V1.1 校准。）
- **单向门覆盖：** 100% 的安全关键决策（`door_type: one-way` OR 分类器标记的动态发现项）以完整技术细节单独浮现。无上限。
- **翻转往返：** 用户输入 `flip test-coverage-bookclub-form`。原始自动接受决策作为 AskUserQuestion 重新打开。用户的新选择持久保存到静默决策块（或如果用户翻到显式浮现则被移除）。
- **每阶段可观测性：** `/plan-tune` 可以显示任何会话每阶段 AskUserQuestion 次数，从 question-log.jsonl 的新 `phase` 字段读取。
- **首次运行减少：** 新用户在第一个真正技能运行之前看到 ≤ 1 个元提示（lake 介绍），对比 V1 的 4 个（lake + 遥测 + 主动 + 路由）。
- **人工重跑：** Louise + Garry 独立定性审查，与 V1 相同模式。

## 对 V1 的依赖

V1.1 建立在 V1 的基础设施之上：
- `explain_level` 配置键 + 序言回显模式（A4）。
- 术语列表 + 写作风格部分（V1.1 的打断语言应尊重 ELI10 规则）。
- V0 休眠负面测试（V1.1 也不会唤醒 5D 心理图谱机械）。
- V1 捕获的 Louise 转录（校准验收标准的基线）。

V1.1 不依赖任何 V2 项（E1 基质接线、叙事/氛围等）。

## 审查计划

- **预工作：** 从当前 V0 数据捕获真实问题日志分布。用作范围 #8 的校准输入。
- **CEO 审查。** 前提挑战：节奏是否正确的修复，还是 V1.1 应考虑完全移除阶段？（例如，将 CEO + Design + Eng + DX 折叠为单次统一评审通过。）范围模式：可能是 SELECTIVE EXPANSION（节奏是核心，相关改进是精选）。
- **Codex 审查。** 对 V1.1 计划的独立通过。预期对控制流改变（范围 #4）进行特别审查，因为那是 V1 挣扎的领域。
- **DX 审查。** 关注翻转机制的用户体验 —— `flip <id>` 是否可发现，命令语法是否自然，错误路径是否清晰。
- **工程审查 ×N。** 预期多次通过与 V1 相同。
