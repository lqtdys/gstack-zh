由 /office-hours + /plan-ceo-review + /plan-eng-review 在 2026-03-28 生成
更新日期：2026-04-01（Session Intelligence 后，经 Codex 审查）
分支：garrytan/ce-features
仓库：gstack
状态：ACTIVE
模式：开源 / 社区

## 问题陈述

GStack 在会话间运行 30+ 个技能，但在它们之间什么都不学习。一个 /review 会话捕获了 N+1 查询模式，而下一个对同一代码库的 /review 从零开始。一个 /ship 运行发现了测试命令，而每个未来的 /ship 重新发现它。一个 /investigate 发现了一个棘手的竞争条件，而未来的会话不知道它。

每个 AI 编码工具都有这个问题。Cursor 有按用户记忆。Claude Code 有 CLAUDE.md。Windsurf 有持久上下文。但它们都没有复利。它们都没有结构化它们学到的东西。它们都没有跨技能共享知识。

## 我们正在构建什么

跨会话和技能复利的按项目制度知识。结构化的、类型化的、信心评分的学习，每个 gstack 技能都可以读写。目标：在同一代码库上 20 个会话后，gstack 知道每个架构决策、每个过去的 bug 模式以及它每次出错的情况。

## 北极星

/autoship（发布 5）。一个完整的工程团队一键完成。描述一个功能，批准计划，其他一切都自动。/autoship 没有学习（R1）、审查质量（R2）、会话持久性（R3）和自适应仪式（R4）就不能工作。发布 1-4 是使 /autoship 真正工作的基础设施。

## 受众

YC 创始人在用 AI 构建。那些每周在真实代码库上运行 gstack 20+ 次并在被问两次同一问题时注意到的人。

## 差异化

| 工具 | 记忆模型 | 范围 | 结构 |
|------|----------|------|------|
| Cursor | 按用户聊天记忆 | 每会话 | 非结构化 |
| CLAUDE.md | 静态文件 | 按项目 | 手动 |
| Windsurf | 持久上下文 | 每会话 | 非结构化 |
| **GStack** | **按项目 JSONL** | **跨会话，跨技能** | **类型化，评分，衰减** |

## 状态系统

gstack 有四个不同的持久层。它们共享存储模式（`~/.gstack/projects/$SLUG/` 中的 JSONL）但服务于不同目的：

| 系统 | 文件 | 它存储什么 | 写入者 | 读取者 |
|------|------|----------|---------|---------|
| **学习** | `learnings.jsonl` | 制度知识（陷阱、模式、偏好） | 所有技能 | 所有技能（序言） |
| **时间线** | `timeline.jsonl` | 事件历史（技能启动/完成、分支、结果） | 序言（自动） | /retro，序言上下文恢复 |
| **检查点** | `checkpoints/*.md` | 工作状态快照（决策、剩余工作、文件） | /checkpoint，/ship，/investigate | 序言上下文恢复，/checkpoint 恢复 |
| **健康** | `health-history.jsonl` | 代码质量分数随时间（每工具，复合） | /health | /retro，/ship（门），/health（趋势） |

这些不重叠。学习 = 你知道什么。时间线 = 发生了什么。检查点 = 你在哪里。健康 = 代码有多好。每个回答不同的问题。

## 发布路线图

### 发布 1："GStack 学习"（v0.13-0.14）—— 已发运

**标题：** 每个会话让下一个更聪明。

已发运内容：
- 在 `~/.gstack/projects/{slug}/learnings.jsonl` 处的学习持久化
- `/learn` 技能用于手动审查、搜索、修剪、导出
- 所有审查发现的置信度校准（带显示规则的 1-10 分数）
- 观察/推断学习的置信度衰减（30 天扣 1 分）
- 跨项目学习发现（选择加入，AskUserQuestion 同意）
- 当审查匹配过去的学习时的"学习应用"标注
- 集成到 /review、/ship、/plan-*、/office-hours、/investigate、/retro

模式：
```json
{
  "ts": "2026-03-28T12:00:00Z",
  "skill": "review",
  "type": "pitfall",
  "key": "n-plus-one-activerecord",
  "insight": "在列表端点中始终检查 includes() 用于 has_many",
  "confidence": 8,
  "source": "observed",
  "branch": "feature-x",
  "commit": "abc1234",
  "files": ["app/models/user.rb"]
}
```

类型：`pattern` | `pitfall` | `preference` | `architecture` | `tool`
来源：`observed` | `user-stated` | `inferred` | `cross-model`

架构：仅追加 JSONL。重复项在读取时解决（按 key+type 的最新获胜者）。无写入时无变异，无竞争条件。

### 发布 2："审查军团"（v0.14.3-0.14.4）—— 已发运

**标题：** 每次 PR 10 个专业审查员。

已发运内容：
- 7 个并行专业子代理：始终开启（测试、可维护性）+ 条件性（安全、性能、数据迁移、API 契约、设计）+ 红队（大 diff/关键发现）
- 带信心分数的结构化 JSON 发现 + 跨代理指纹去重
- PR 质量分数（0-10）按审查记录 + /retro 趋势
- 学习知情的专家提示，按领域注入过去的陷阱
- 多专家共识高亮，确认的发现得到增强
- 通过 PLAN_COMPLETION_AUDIT 增强交付完整性
- 检查表重构：CRITICAL 类别保留在主要通道中，专家类别提取到 review/specialists/ 中的焦点检查表

### 发布 2.5："审查军团扩展" —— 尚未发运

**标题：** 在 R2 证明稳定后发运。检查核心循环的表现如何。

预检查：审查 R2 质量指标（PR 专业分数、专业命中率、误报率、E2E 测试稳定性）。如果核心循环有问题，先修复那些。

发运内容：
- E1：自适应专家门，自动跳过 0-发现记录的专业。通过 gstack-learnings-log 存储按项目命中率。用户可以用 --security 等强制。
- E3：测试桩生成，每个专业在发现旁输出 TEST_STUB。从项目（Jest/Vitest/RSpec/pytest/Go test）。流入 Fix-First：AUTO-FIX 应用修复 + 创建测试文件。
- E5：跨审查发现去重，读取 gstack-review-read 的先验审查条目。抑制匹配先前用户跳过发现的发现。
- E7：专家性能跟踪，通过 gstack-review-log 记录每专业指标。时间线集成：专业运行出现在 timeline.jsonl 中用于 /retro 趋势。

### 发布 3："会话智能"（v0.15.0）—— 已发运

**标题：** 你的 AI 会话记住发生了什么。

已发运内容：
- 会话时间线：每个技能自动记录启动/完成事件到 `~/.gstack/projects/$SLUG/timeline.jsonl`。仅本地，永不发送到任何地方，无论遥测设置如何始终开启。
- 上下文恢复：压缩或会话启动后，序言列出最近的 CEO 计划、审查和检查点。代理读取最新的一个来恢复上下文。
- 跨会话注入：序言为当前分支打印 LAST_SESSION 和 LATEST_CHECKPOINT。你在输入任何内容之前就能看到你离开的地方。
- 预测性技能建议：如果你最近的 3 个会话遵循模式（review，ship，review），gstack 建议你可能接下来想要什么。
- "欢迎回来"上下文在会话开始时合成消息。
- `/checkpoint` 技能：保存/恢复/列出工作状态快照。跨分支列出用于 Conductor 工作区间代理交接。
- `/health` 技能：代码质量记分卡包装项目工具（tsc、biome、knip、shellcheck、测试）。复合 0-10 分数，趋势跟踪，分数下降时的改进建议。
- 时间线二进制：`bin/gstack-timeline-log` 和 `bin/gstack-timeline-read`。
- 路由规则：/checkpoint 和 /health 添加到序言技能路由。

设计文档：`docs/designs/SESSION_INTELLIGENCE.md`

### 发布 4："自适应仪式" —— 尚未发运

**标题：** GStack 尊重你的时间而不牺牲安全。

仪式和信任是独立的关注。仪式 = PR 经历的审查/测试/QA 步骤集。信任 = 确定适用什么仪式级别的政策引擎。它们相互作用但不合并。

发运内容：

**仪式级别：**
- FULL：所有专业、对抗、Codex 结构化审查、覆盖审计、计划完成。用于大 diff、新功能、迁移、auth 更改。
- STANDARD：对抗 + Codex、覆盖审计、计划完成。用于中等 diff，典型功能工作。
- FAST：仅对抗。用于可信项目上的小型、经过良好测试的更改。

**信任策略引擎：**
- 范围感知信任。信任按更改类别获得，而非全局。文档上的干净历史不会在迁移 PR 上买到信任。
- 更改类别检测：文档、测试、配置、前端、后端、迁移、auth、基础设施。每个类别有自己的信任阈值。
- 信任信号：连续干净审查（按类别）、/health 分数稳定性、回归频率、测试覆盖率趋势。
- 信任永不开绿灯：迁移、auth/权限更改、新 API 端点、基础设施更改。这些无论信任级别如何始终获得 FULL 仪式。
- 渐进降级，而非二进制重置。单个回归不会重置所有信任。它按降级一个级别对该更改类别的信任进行评估。

**范围评估：**
- /review、/ship、/autoplan 中的 TINY/SMALL/MEDIUM/LARGE 分类基于 diff 大小、触及的文件和更改类别。
- 仪式级别 = f(范围，信任，更改类别)。

**TODO 生命周期：**
- /triage 用于传入 TODO 的交互式批准
- /resolve 通过并行代理批量解决

### 发布 5："/autoship —— 一键，全功能" —— 尚未发运

**标题：** 描述一个功能。批准计划。其他一切都自动。

/autoship 是一个可恢复状态机，而非线性管道。审查和 QA 可以发送工作回到构建/修复。压缩可以中断任何阶段。系统必须优雅恢复。

```
                    │  START   │
                    └────┬─────┘
                         │
                    ┌────▼─────┐
                    │ /office- │
                    │  hours   │
                    └────┬─────┘
                         │
                    ┌────▼─────┐
                    │/autoplan │ ◄── 单次批准门
                    └────┬─────┘
                         │
              ┌──────────▼──────────┐
              │       BUILD         │ ◄── /checkpoint 自动保存
              └──────────┬──────────┘
                         │
              ┌──────────▼──────────┐
              │      /health        │ ◄── 质量门
              │   (score >= 7.0)    │
              └──────────┬──────────┘
                         │ fail → 回退到 BUILD
              ┌──────────▼──────────┐
              │      /review        │
              └──────────┬──────────┘
                         │ ASK 项目 → 回退到 BUILD
              ┌──────────▼──────────┐
              │        /qa          │
              └──────────┬──────────┘
                         │ 发现 bug → 回退到 BUILD
              ┌──────────▼──────────┐
              │       /ship         │
              └──────────┬──────────┘
                         │
              ┌──────────▼──────────┐
              │ /checkpoint archive │ ◄── 保留，不销毁
              └─────────────────────┘
```

发运内容：
- 具有上述状态机的 /autoship 自主管道。
  每个阶段写入 timeline.jsonl。检查点在阶段之间自动保存。
  压缩恢复：上下文恢复读取检查点 + 时间线，在最后完成的阶段恢复。
- 完成时检查点归档（非删除）。恢复状态为调试失败的 autoship 运行保留。
- /ideate 头脑风暴技能（并行发散代理 + 对抗过滤）
- /plan-eng-review 中的研究代理（代码库分析师、历史分析师、最佳实践研究员、学习研究员）

依赖：R1（研究代理的学习）、R2（质量的审查军团），R3（持久性的会话智能），R4（速度的自适应仪式）。

## 风险登记

### 代理信号作为跳过审查的许可
（由 Codex 审查发现，2026-04-01）

/health 分数、干净的审查历史和时间线模式是有用的信号。它们不是安全的证明。如果这些信号既供给仪式减少又供给 /autoship，故障模式是罕见的、沉默的、高严重性的错误。缓解：
- 某些更改类别永不开绿灯（迁移、auth、基础安排、新端点）。
- 信任降级是渐进的，而非二进制重置。
- /autoship 始终在每个项目的首次运行上运行 FULL 信任。信任是赚来的。

### 陈旧上下文恢复
（由 Codex 审查发现，2026-04-01）

上下文恢复可注入错误分支状态、过时计划或无效检查点。缓解：
- 检查点在 YAML frontmatter 中包含分支名称。上下文恢复按当前分支过滤。
- 时间线 grep 在显示 LAST_SESSION 之前按分支过滤。
- 陈旧工件检测：如果检查点 >7 天旧，标注为潜在陈旧而非呈现为当前。

### 需要验证指标
（由 Codex 审查发现，2026-04-01）

在发运 R4（自适应仪式）之前，测量：
- 预测建议准确性（用户是否运行了建议的技能？）
- 信任策略错误跳过率（绿灯 PR 是否有合并后问题？）
- 上下文恢复准确性（恢复的上下文是否匹配实际状态？）
- /health 分数与实际代码质量的相关性（高分是否预测更少的生产 bug？）

这些指标应在 R3 使用期间收集并在 R4 发运前审查。
