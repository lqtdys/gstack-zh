# 为什么 gstack 在文档中使用 Diataxis 框架

gstack 中的两个文档技能 — `/document-release` 与 `/document-generate` — 两者都讲 Diataxis。新实体按四个象限打分。覆盖缺口按象限标签出现在 PR 正文中。本文解释这种词汇表为何具有承载作用，以及简单的"就写 markdown"这种方案在 gstack 的运营规模下为何会崩溃。

## 问题

文档腐烂是最容易被忽视的腐烂类型。代码停止编译，你会立即注意到。测试失败，CI 会尖叫。文档默默腐坏 — README 仍然能解析、安装命令仍然能复制粘贴 — 唯一的信号是数周后某个困惑的用户提 issue 或默默离开。

gstack 有 45 个以上的技能。每个技能都是 SKILL.md 加上 `.tmpl` 模板加上（理想情况下）入门教程以及它为何这样运作的解释。乘以同样拥有类似表面积的 gstack 用户数量，维护负担是显而易见的。

第一种天真失败模式是"每个团队按自己的格式写文档"。一个项目有 Wiki。另一个有嵌套 README 文件。第三个只有参考文档，没有教程。第四个的教程已经编译不通过了。你没法写工具来跨所有这些都做审计，因为没有共享的词汇表来定义什么是好的覆盖。

第二种失败模式更微妙：即使一个团队很自律，他们往往写那种匹配当前心态的文档。构建模式下的工程师写参考文档。发布模式下写教程。维护模式下写 troubleshooting how-to。没人今天醒来会说"今天我来解释文档，说明为什么选择这个架构" — 所以解释腐烂的积累速度最快。

## 方法

Diataxis（Daniele Procida，最初在 Divio，现在被 CPython、Django、NumPy、FastAPI、GitHub 文档等采用）根据**读者意图**将文档分为四个象限：

```
                    理论性                        实践性
                    (理解)                        (行动)

  学习             +-----------------------------+----------------------------+
  (研究)           |                             |                            |
                   |   解释                      |   教程                     |
                   |   "X 为什么存在？"          |   "带我一步步完成 X        |
                   |                             |   第一次"                  |
                   |   讨论代码                  |   教授代码                 |
                   +-----------------------------+----------------------------+

  工作             +-----------------------------+----------------------------+
  (使用)           |                             |                            |
                   |   参考                      |   操作指南（How-to）       |
                   |   "Y 的确切签名是什么？"    |   "如何用 X 完成 Y？"      |
                   |   "                         |                            |
                   |   描述代码                  |   使用代码                 |
                   +-----------------------------+----------------------------+
```

教程模式的读者通过动手学习。他们想要一条有保障成功的手把手引导路径。操作指南模式的读者已经了解基础知识，想要特定任务的配方。参考模式的读者希望获取准确、完整的事实表覆盖。解释模式的读者想要理解一个设计决策。

同一个人在不同时间从这些模式阅读项目。同一段文字不能同时服务所有四种 — 教程需要的手把手引导会让参考读者觉得慢；参考需要的完整性会让教程读者觉得过载。

## 为什么这作为覆盖透镜很重要

以 Diataxis 术语编写的覆盖地图给你一个确定性的回答："文档更新了吗？" — 不是"有没有 README"而是"这个新技能有没有教程、常见任务有没有操作指南、API 有没有参考文档、不显而易见的设计选择有没有解释文档？"

`/document-release` 步骤 1.5 遍历差异，提取新的公开表面（技能、CLI 标志、配置选项、API 端点），并为每个实体跨四个象限打分。零覆盖的项目成为**关键缺口**。仅有参考覆盖的项目（gstack 自身历史上最常见的失败模式）成为**常见缺口**。两者都落在 PR 正文中供审查者查看。

`/document-generate` 刻意按四个象限写文档。它拒绝混合：教程不会有"配置"段落，参考文档不会有"你将构建什么"的段落。该技能的 9 个步骤按参考 → 解释 → 操作指南 → 教程的顺序，因为这种顺序匹配依赖关系：参考固定词汇，解释证明设计合理性，操作指南在两者之上构建，教程最后且最难。

## 取舍

**Diataxis 增加了读者必须学习的词汇。**从未听说过"参考与解释"的用户起初可能觉得这些标签奇怪。缓解措施是 Diataxis 标签一旦见过一次就清晰明了，而且标签永远不会出现在文档本身中 — 它们出现在覆盖地图和 PR 正文中，只在审查者那里出现，终端用户看不到。

**四个文件而非一个。**一个小型技能可能只有一个混合四种模式的 `docs/SKILL.md` 文件。Diataxis 将其拆分为四个。缓解措施：AI 生成让四文件结构变得便宜，象限之间的交叉链接是机械式的（每个参考文档链接到其操作指南，每个操作指南链接到其参考文档，等等），可审计性收益巨大 — `/document-release` 可以自动打分覆盖度。

**Diataxis 不是唯一好的框架。**"每页都是首页"（Mark Baker）、*Write the Docs* 社区的四种文档类型、Google 开发者文档风格指南 — 都有不同的切分方式。gstack 选择 Diataxis 因为有最强的外部采用率（CPython、Django、NumPy、FastAPI 等），这意味着下游用户最有可能之前见过这些词汇，并且象限标签能干净利落地映射为覆盖地图信号。

## 考虑过的替代方案

**"就写 README 段落。"**在 gstack 历史上隐式尝试过。失败模式：教程在 README 中累积直到 README 超过 800 行并且没人读第 50 行之后的内容。Diataxis 将它们拆分为独立文件，每个都可从 README 目录中发现。

**自定义内部分类法。**看似可以量身定制。被拒绝，因为每个团队都会发明自己的词汇，`/document-release` 会失去其跨项目审计能力。Diataxis 是通用语。

**仅自动生成参考文档。**通过 JSDoc / TypeDoc / Sphinx 等工具在很多项目中尝试过。没有解释的参考文档对新来者变得不可逾越；没有教程，API 上手困难。参考是必要但不充分的。

**没有文档框架，全靠直觉。**大多数项目的现状。静默失败 — 用户选择离开而不是提 issue，所以反馈循环断裂。Diataxis 在用户抱怨之前就给出结构化信号。
