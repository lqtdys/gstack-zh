## `/office-hours`

这是每个项目应该开始的地方。

在你计划、审查、编写代码之前 — 与 YC 风格的合伙人坐下来，思考你实际上在构建什么。不是你认为你在构建什么。你*实际上*在构建什么。

### 重新框架

这是在一个真实项目中发生的事情。用户说："我想为我的日历构建一个每日简报应用。"合理的要求。然后它询问了痛苦 — 具体例子，不是假设的。他们描述了一个遗漏事情的助理、跨多个 Google 账户的日历项目伴有陈旧信息、AI 生成的准备文档、带有错误位置的事件需要很长时间才能追踪。

它回来说：*"我要挑战这个框架，因为我认为你已经超越了它。你说的是'多 Google 日历管理的每日简报应用。'但你实际上描述的是一个个人参谋长 AI。"*

然后它提取了用户没有意识到他们正在描述的五种能力：

1. **看你日历**跨所有账户并检测陈旧信息、缺失位置、权限差距
2. **生成真正准备的工作** — 不是物流摘要，而是为董事会、播客、筹款活动做准备的*智力工作*
3. **管理你的 CRM** — 你在见谁、关系是什么、他们想要什么、历史是什么
4. **优先你的时间** — 标记何时需要提前开始准备、主动安排时间块、按重要性排列事件
5. **用金钱换杠杆** — 积极寻找委派或自动化的方法

那个重新框架改变了整个项目。他们即将构建一个日历应用。现在他们正在构建有价值十倍的东西 — 因为技能听了他们的痛苦而不是他们的功能请求。

### 前提挑战

在重新框架之后，它呈现前提供您验证。不是"这个听起来好吗？" — 关于产品的实际可证伪的主张：

1. 日历是锚数据源，但价值在于其上的智能层
2. 助理不会被替换 — 他们会被超级赋能
3. 最窄的楔形是一个真正有效的每日简报
4. CRM 集成是必须具备的，不是可有可无的

你同意、不同意或调整。你接受的每个前提都在设计文档中承重。

### 实现替代方案

然后它生成 2-3 个具体的实现方法，带有诚实的effort估算：

- **方法 A：每日简报优先** — 最窄的楔形、明天发货、M 努力（人类：~3 周 / CC：~2 天）
- **方法 B：CRM 优先** — 首先构建关系图、L 努力（人类：~6 周 / CC：~4 天）
- **方法 C：完全愿景** — 一切同时进行、XL 努力（人类：~3 个月 / CC：~1.5 周）

推荐 A，因为你从真实使用中学习。CRM 数据在第二周自然到来。

### 两种模式

**创业模式** — 为创始人及内部创业者构建业务。你得到六个从 YC 合伙人评估产品方式中提炼的强制问题：需求现实、现状、绝望的具体性、最窄楔形、观察和惊喜、未来适应性。这些问题是故意令人不舒服的。如果你不能说出需要你产品的具体人类，那是在写任何代码之前需要学习的最重要的事情。

**建设者模式** — 用于黑客松、副项目、开源、学习和娱乐。你得到一个热情的合作者，帮助你找到你想法的最酷版本。什么会让某人说"哇"？最快路径到你可以分享的东西是什么？这些问题是生成性的，不是审问性的。

### 设计文档

两种模式都以写入 `~/.gstack/projects/` 的设计文档结束 — 该文档直接输入到 `/plan-ceo-review` 和 `/plan-eng-review`。完整的生命周期现在是：`office-hours → 计划 → 实现 → 审查 → QA → 发布 → 回顾`。

在设计文档被批准后，`/office-hours` 反思它注意到了你思考方式的什么 — 不是通用的赞美，而是对你在会话期间说过的具体事情的回调。这些观察也出现在设计文档中，所以你在以后重新阅读时会重新遇到它们。

## `/plan-ceo-review`

这是我的**创始人模式**。

这是我想让模型有品味、雄心、用户同理心和长远视野思考的地方。我不想要它字面上接受请求。我想它先问一个更重要的问题：

**这个产品实际上是为了什么？**

我认为这是**Brian Chesky 模式**。

关键不是实现显而易见的票。关键是重新从用户角度思考问题并找到那个感觉不可避免、令人愉悦、甚至有点魔幻的版本。

### 示例

假设我正在构建一个 Craigslist 风格的列表应用，我说：

> "让卖家为他们商品上传一张照片。"

一个弱的助手会添加一个文件选择器并保存图像。

那不是真正的产品。

在 `/plan-ceo-review` 中，我想让模型问"照片上传"是否真的是功能。也许真正的功能是帮助某人创建一个真正能卖出的列表。

如果那是真正的工作，整个计划就会改变。

现在模型应该问：

* 我们能否从照片中识别产品？
* 我们能否推断 SKU 或型号？
* 我们能否搜索网络并自动起草标题和描述？
* 我们能否拉取规格、类别和定价比较？
* 我们能否建议哪张照片作为主图转换最佳？
* 我们能否检测上传的照片是否丑陋、黑暗、杂乱或低信任？
* 我们能否让体验感觉优质，而不是像 2007 年的死表单？

那就是 `/plan-ceo-review` 为我做的。

它不仅问，"我如何添加这个功能？"
它问，**"这个请求中隐藏的 10 星产品是什么？"**

### 四种模式

- **范围扩展** — 大梦想。代理提出雄心勃勃的版本。每个扩展都作为您选择加入的单独决定呈现。热情推荐。
- **选择性扩展** — 将您当前范围作为基线，但看看还有什么可能。代理逐一呈现中立推荐的机会 — 你挑选值得做的。
- **持有范围** — 对现有计划的最大严谨性。没有扩展呈现。
- **范围缩减** — 找到最小可行版本。削减其他所有。

愿景和决策持久化到 `~/.gstack/projects/` 以便它们超越对话。卓越的愿景可以提升到您仓库中的 `docs/designs/` 以供团队使用。

## `/plan-eng-review`

这是我的**工程经理模式**。

一旦产品方向对了，我想要完全不同的智能。我不要更多蔓延式的创意。我不要更多"如果不是很酷吗。"我想模型成为我最好的技术负责人。

这个模式应该搞定：

* 架构
* 系统边界
* 数据流
* 状态转换
* 故障模式
* 边界情况
* 信任边界
* 测试覆盖

和一个对我来说惊人的解锁：**图**。

当你强迫 LLM 绘制系统时，它们会变得更加完整。序列图、状态图、组件图、数据流图、甚至测试矩阵。图强制隐藏假设暴露出来。它们让模糊的规划变得更加困难。

所以 `/plan-eng-review` 是我想要模型构建能够承载产品愿景的技术脊柱的地方。

### 示例

拿同样的列表应用示例。

假设 `/plan-ceo-review` 已经完成了它的工作。我们决定真正不只是照片上传。它是一个智能列表流程：

* 上传照片
* 识别产品
* 从网络丰富列表
* 起草强有力的标题和描述
* 建议最佳主图

现在 `/plan-eng-review` 接手。

现在我想让模型回答这样的问题：

* 上传、分类、丰富和起草生成的架构是什么？
* 哪些步骤同步发生，哪些进入后台作业？
* 应用服务器、对象存储、视觉模型、搜索/丰富 API 和列表数据库之间的边界在哪里？
* 如果上传成功但丰富失败怎么办？
* 如果产品识别是低置信度怎么办？
* 重试如何工作？
* 我们如何防止重复作业？
* 什么持久化，什么可以安全地重新计算？

这就是我想要图的地方 — 架构图、状态模型、数据流图、测试矩阵。图强制隐藏假设暴露出来。它们让模糊的规划变得更加困难。

那就是 `/plan-eng-review`。

不是"让想法更小。"
**让想法可构建。**

## `/plan-design-review`

这是我的**高级设计师审查你的计划** — 在你写一行代码之前。

大多数计划描述了后端做什么，但从不指定用户实际看到什么。空状态？错误状态？加载状态？移动端布局？AI 垃圾风险？这些决定被推迟到"实施期间搞清楚" — 然后一个工程师因为没人指定更好的东西而把"没有找到项目。"作为空状态发布。

`/plan-design-review` 在计划期间捕获所有这一切，当修复成本很低时。

它就像 `/plan-ceo-review` 和 `/plan-eng-review` 一样工作 — 交互式的，一次一个问题，带有 **STOP + AskUserQuestion** 模式。它将每个设计维度评分 0-10，解释 10 分是什么，然后编辑计划以达到那里。评分驱动工作：低评分 = 大量修复，高评分 = 快速通过。

计划上的七轮：信息架构、交互状态覆盖、用户旅程、AI 垃圾风险、设计系统对齐、响应式/可访问性，以及未解决的设计决策。对于每一轮，它找到差距并直接修复（明显的）或要求您做出设计选择（真正的权衡）。

### 示例

```
您：   /plan-design-review

Claude： 初始设计评分：4/10

        "这个计划描述了一个用户仪表板但从不指定用户首先看到什么。它说'带有图标的卡片' — 看起来像每个 SaaS 模板。它提到零加载状态、零空状态和零移动端行为。"

        Pass 1 (信息架构)：3/10
        "一个 10 分将为每个屏幕定义主/次/第三内容层次。"
        → 添加了信息层次部分到计划

        Pass 2 (交互状态)：2/10
        "计划有 4 个 UI 功能但指定了 20 个交互状态中的 0 个（4 个功能 × 每个 5 个状态）。"
        → 添加了交互状态表到计划

        Pass 4 (AI 垃圾)：4/10
        "计划说'干净、现代的 UI 带有卡片和图标'和'带有渐变的主角部分。'这些是前 2 个看起来像 AI 生成的模式。"
        → 用具体的、有意的替代方案重写了 UI 描述

        总分：4/10 → 10 分修复后
        "计划是设计完整的。实施后运行 /design-review 进行视觉 QA。"
```

当您重新运行它时，已经 8+ 以上的部分获得快速通过。低于 8 的部分获得完整治疗。对于实施后的实时站点视觉审查，使用 `/design-review`。

## `/design-consultation`

这是我的**设计合伙人模式**。

`/plan-design-review` 审查已存在的站点。`/design-consultation` 适用于当你什么都没有时 — 没有设计系统、没有字体选择、没有调色板。你从零开始，你想要一个高级设计师坐下来与你一起构建整个视觉身份。

这是一个对话，不是表单。代理询问您的产品、用户和受众。它思考您的产品需要沟通什么 — 信任、速度、工艺、温暖，任何适合的 — 然后从那里逆向工作到具体的然后它提出一个完整、连贯的设计系统：美学方向、排版（3+ 种字体带有具体角色）、带有十六进制值的调色板、间距比例、布局方法和动作策略。每个推荐都有理由。每个选择都加强其他选择。

但连贯性是底线。每个开发者工具仪表板看起来都一样 — 干净的无衬线、柔和的灰色、蓝色强调。它们都是连贯的。它们都是可遗忘的。一个看起来"不错"的产品和人们真正识别的产品之间的区别是**深思熟虑的创意风险**：标题的意想不到的无衬线、你的类别中没有人使用的粗体强调、更紧密的间距让您的数据感觉权威而不是空灵。

那就是 `/design-consultation` 真正的意义。它不仅提出安全的系统。它提出安全的选择和风险 — 并告诉您哪个是哪个。"这里是在你的类别中保持你识字的选择。以及这里我认为你应该打破常规的地方，为什么。"您挑选要承担的风险。代理检查整个系统是否仍然连贯。

如果愿意，代理已经存在的内容在您进行竞争空间中 — 拍摄真实站点的屏幕截图、分析它们的字体、颜色和间距 — 以便您可以看到在选择之前的景观。这不是复制。这是进入球场以便知道哪些惯例值得打破。

在您同意系统后，它会生成一个交互式 HTML 预览页面 — 不仅仅是色板和字体样本，而是逼真的产品页面。如果您正在构建仪表板，您会看到一个带有侧栏、数据表格和统计卡片的仪表板。如果您正在构建营销站点，您会看到一个带有真实文案和 CTA 的主角部分。所有内容都在您的设计系统中渲染，带有您的产品名称，明暗模式。你在写一行生产代码之前看到你的产品可能感觉如何。

然后它将 `DESIGN.md` 写入您仓库的根目录 — 您的项目的设计真相来源 — 并更新 `CLAUDE.md` 以便每个未来的 Claude Code 会话尊重系统。从那时起，`/design-review` 可以审查它，并且任何处理前端的代理都知道规则。

### 示例

```

Claude： 从 README 看，这看起来像是一个面向工程团队的分析
        仪表板。听起来对吗？告诉我你想要的感觉 —
        你想先让我看看外面有什么以便我们看到格局吗？

您：    是的，这是给工程经理的。给我展示外面的。

Claude： [浏览 Linear、Vercel、Datadog、Posthog — 截图]

        这就是格局。他们都会聚在干净的无衬线
        和柔和的蓝灰色调上。Linear 以暗色优先和
        紧密间距脱颖而出。其余感觉可互换。

        这是我的提议 — 以及我会安全
```

## `/design-review`

这是我的**会编码的设计师模式**。

`/plan-design-review` 在实施前审查您的计划。`/design-review` 在之后审计和修复实时站点。

它运行您实时站点的 80 项视觉审查，然后进入修复循环：对于每个设计发现，它定位源文件，进行最小的 CSS/样式更改，提交 `style(design): FINDING-NNN`，重新导航验证，并拍摄前后截图。每个修复一个提交，完全可二分。

自调节启发式方法针对设计工作进行调优 — CSS 更改获得免费通行证（它们本质上是安全和可逆的），但对组件 JSX/TSX 文件的更改计入风险预算。30 个修复硬上限。如果风险分数超过 20%，它会停止并询问。

### 示例

```
您：   /design-review https://myapp.com

Claude： [运行实时站点的完整 80 项视觉审查]
        设计分数：C  |  AI 垃圾分数：D
        12 个发现（4 高、5 中、3 打磨）

        修复 9 个设计问题...

        style(design): FINDING-001 — 用非对称布局替代 3 列图标网格
        style(design): FINDING-002 — 添加标题比例 48/32/24/18/16
        style(design): FINDING-003 — 移除渐变主角，使用粗体排版
        style(design): FINDING-004 — 为标题添加第二种字体
        style(design): FINDING-005 — 按元素角色变化 border-radius
        style(design): FINDING-006 — 正文左对齐，居中保留给标题
        style(design): FINDING-007 — 为所有交互元素添加 hover/focus 状态
        style(design): FINDING-008 — 添加 prefers-reduced-motion 媒体查询
        style(design): FINDING-009 — 设置最大内容宽度为 680px 用于正文

        最终审查：
        设计分数：C → B+  |  AI 垃圾分数：D → A
        已应用 9 个修复（8 个已验证，1 个尽力）。3 个延迟。

        [带有前后截图的报告已保存到 .gstack/design-reports/]
```

九个提交，每个触及一个关注点。AI 垃圾分数从 D 到 A 因为三个最具辨识度的模式（渐变主角、3 列网格、统一半径）消失了。

## `/design-shotgun`

这是我的**设计探索模式**

你知道那种感觉。你有一个功能、一个页面、一个登陆屏幕……你不确定它应该是什么样子。你可以向 Claude 描述并得到一个答案。但一个答案意味着一个视角，设计是游戏。你需要看到选项。

`/design-shotgun` 使用 GPT Image API 生成 3 个视觉设计变体、在您的浏览器中打开比较板，并等待您的反馈。您选择一个方向、要求更改，或要求全新变体。该板支持重新混合、重新生成和批准操作。

### 循环

1. 您描述想要的（或指向现有页面）
2. 技能读取您的 `DESIGN.md` 以获取品牌约束（如果存在）
3. 它生成 3 个不同的设计变体作为 PNG
4. 比较板在您的浏览器中并排打开所有 3 个
5. 在您喜欢的一个上点击"批准"，或为另一轮提供反馈
6. 批准的变体保存到 `~/.gstack/projects/$SLUG/designs/` 并带有 `approved.json`

那个 `approved.json` 是一种输入 `/design-html` 的方式。设计管道链接：shotgun 选择方向，design-html 将其渲染为工作代码。但 `/design-html` 也与 CEO 计划、设计审查或仅描述配合工作。

### 口味记忆

技能记住您的偏好在会话之间。如果您一直喜欢极简设计而不是繁忙的设计，它会偏向未来的生成。这不是您配置的……它从您的批准中出现。

### 示例

```
您：   /design-shotgun — 开发者工具登陆页的主角部分

Claude： [生成 3 个变体]
        变体 A：粗体排版、深色背景、代码片段主角
        变体 B：分体布局、产品截图左、文案右
        变体 C：极简、居中标题、渐变强调

        [在 localhost:PORT 打开比较板]

您：    [在浏览器中点击变体 A 的"批准"]

Claude： 已批准变体 A。保存到 ~/.gstack/projects/myapp/designs/
        下一步：运行 /design-html 从这个模型生成生产 HTML。
```

## `/design-html`

这是我的**设计到代码模式**。

每个 AI 代码生成工具生成静态 CSS。硬编码高度。调整大小时文本溢出。断点突然折断而不是流动。输出在一个视口大小看起来对并且在其他每个都出错。

`/design-html` 修复了这一点。它使用 [Pretext](https://github.com/chenglou/pretext)（由 Cheng Lou（前 React 核心，Midjourney 前端）开发）生成 HTML。Pretext 是一个 15KB 的库，它计算文本布局而不进行 DOM 测量。文本重新流动。高度根据内容调整。卡片自我调整大小。聊天气泡收缩包装。All sub-millisecond, all dynamic.

它与多个输入源配合工作：来自 `/design-shotgun` 的批准模型、来自 `/plan-ceo-review` 的 CEO 计划、来自 `/plan-design-review` 的设计审查上下文、您提供的 PNG，或只是您想要的描述。它检测什么上下文存在并询问您想如何进行。

### 智能 API 路由

不是每个页面都需要完整的 Pretext 引擎。技能读取设计并选择合适的工具：

- **简单布局**（登陆、营销）：`prepare()` + `layout()` 用于调整大小感知高度
- **卡片网格**（仪表板、列表）：`prepare()` + `layout()` 用于自调整大小的卡片
- **聊天 UI**：`walkLineRanges()` 用于紧密贴合的气泡带有零浪费像素
- **编辑布局**：`layoutNextLine()` 用于文本围绕障碍物流动
- **复杂编辑**：完整引擎带 `layoutWithLines()` 用于手动线渲染

### 精炼循环

1. 从 `approved.json` 读取批准的模型
2. 使用 GPT-4o 视觉提取实现规格（颜色、排版、布局）
