# 落地前审查清单

## 说明

审查以下 `git diff origin/main` 输出是否有下列问题。要具体 — 引用 `file:line` 并提出修复建议。跳过任何没问题的部分。只标记真正的问题。

**两轮审查：**
- **第一轮（严重）：** 先运行 SQL 与数据安全、竞态条件与并发、LLM 输出信任边界、Shell 注入、枚举完整性。最高严重性。
- **第二轮（信息性）：** 运行剩余类别。较低严重性，但同样会被处理。
- **专业类别（由并行子代理处理，不在本清单中）：** 测试缺口、死代码、魔术数字、条件副作用、性能与捆包影响、密码与熵。参见 `review/specialists/`。

所有发现都通过"先修复审查"进行处理：明显机械化的修复自动应用，真正模糊的问题打包成单个用户问题。

**输出格式：**

```
Pre-Landing Review: N 个问题（X 严重，Y 信息性）

**AUTO-FIXED:**
- [file:line] 问题 → 已应用修复

**NEEDS INPUT:**
- [file:line] 问题描述
  推荐修复：建议的修复
```

如果未发现问题：`Pre-Landing Review: No issues found.`

要简洁。每个问题：一行描述问题，一行给修复。不要前言、不要总结、不要"总体来看还不错"。

## 审查类别

### 第一轮 — 严重

#### SQL 与数据安全
- SQL 中的字符串插值（即使值是 `.to_i`/`.to_f` — 使用参数化查询（Rails：sanitize_sql_array/Arel；Node：预处理语句；Python：参数化查询））
- TOCTOU 竞态：应该是原子 `WHERE` + `update_all` 的检查-然后-设置模式
- 绕过模型验证直接进行 DB 写入（Rails：update_column；Django：QuerySet.update()；Prisma：原始查询）
- N+1 查询：在循环/视图中使用的关联缺少预加载（Rails：.includes()；SQLAlchemy：joinedload()；Prisma：include）

#### 竞态条件与并发
- 读取-检查-写入没有唯一性约束或捕获重复键错误并重试（例如 `where(hash:).first` 然后 `save!` 没有处理并发插入）
- 无唯一 DB 索引的 find-or-create — 并发调用可能创建重复项
- 不使用原子 `WHERE old_status = ? UPDATE SET new_status` 的状态转换 — 并发更新可能跳过或双重应用转换
- 在用户可控数据上不安全渲染 HTML（Rails：.html_safe/raw()；React：dangerouslySetInnerHTML；Vue：v-html；Django：|safe/mark_safe）导致 XSS

#### LLM 输出信任边界
- LLM 生成的值（电子邮件、URL、姓名）写入 DB 或传递给邮件程序而没有格式验证。在持久化前添加轻量守卫（`EMAIL_REGEXP`、`URI.parse`、`.strip`）。
- 结构化工具输出（数组、哈希）在数据库写入前没有类型/形状检查。
- 无白名单地抓取 LLM 生成的 URL — 如果 URL 指向内部网络则有 SSRF 风险（Python：`urllib.parse.urlparse` → 在 `requests.get`/`httpx.get` 前检查主机名是否在阻止列表中）
- LLM 输出存储在知识库或向量 DB 中没有清理 — 存储的提示注入风险

#### Shell 注入（Python 专用）
- `subprocess.run()` / `subprocess.call()` / `subprocess.Popen()` 使用 `shell=True` 且在命令字符串中有 f-string/`.format()` 插值 — 改为使用参数数组
- `os.system()` 使用变量插值 — 替换为使用参数数组的 `subprocess.run()`
- `eval()` / `exec()` 在 LLM 生成的代码上运行没有沙箱隔离

#### 枚举与值完整性
当 diff 引入新的枚举值、状态字符串、tier 名称或类型常量时：
- **追踪每个消费者。** 阅读（不要只是 grep — 要阅读）每个切换、过滤或显示该值的文件。如果任何消费者没有处理新值，标记它。常见遗漏：在前端下拉列表添加了一个值，但后端模型/计算方法没有持久化它。
- **检查白名单/过滤数组。** 搜索包含同级值（如添加 "revise" 到 tiers 则找到每个 `%w[quick lfg mega]`）的数组或 `%w[]` 列表，并验证在需要的地方包含 "revise"。
- **检查 `case`/`if-elsif` 链。** 如果现有代码在枚举上分支，新值是否会落入错误的默认值？
要执行此操作：使用 Grep 查找所有同级值（如 "lfg" 或 "mega"）的引用来查找所有消费者。阅读每个匹配。此步骤需要阅读 diff 之外的代码。

### 第二轮 — 信息性

#### 异步/同步混合（Python 专用）
- 在 `async def` 端点中使用同步 `subprocess.run()`、`open()`、`requests.get()` — 阻塞事件循环。改用 `asyncio.to_thread()`、`aiofiles` 或 `httpx.AsyncClient`。
- 在异步函数中使用 `time.sleep()` — 使用 `asyncio.sleep()`
- 异步上下文中的同步 DB 调用没有 `run_in_executor()` 包装

#### 列/字段名称安全
- 验证 ORM 查询中的列名（`.select()`、`.eq()`、`.gte()`、`.order()`）与实际 DB 材质对比 — 错误的列名会静默返回空结果或抛出被吞掉的错误
- 检查结果上的 `.get()` 调用使用实际选择的列名
- 有可用的材质文档时对照检查

#### 死代码与一致性（仅版本/changelog — 其他项目由可维护性专家处理）
- PR 标题和 VERSION/CHANGELOG 文件之间的版本不匹配
- 描述更改不准确的 CHANGELOG 条目（例如，"从 X 改为 Y"而 X 从未存在过）

#### LLM 提示问题
- 提示中的 0 索引列表（LLM 可靠地返回 1 索引）
- 列出可用工具/能力的提示文本与 `tool_classes`/`tools` 数组中实际配置的不匹配
- 在多处声明的单词/token 限制可能漂移

#### 完整性缺口
- 完整版本只需 <30 分钟 CC 时间就可以实现的快捷实现（例如，部分枚举处理、不完整的错误路径、容易添加的缺失边缘情况）
- 选项只展示人工团队努力估算 — 应同时展示人工和 CC+gstack 时间
- 添加缺失测试是"湖"不是"海洋"的测试覆盖缺口（例如，缺少负面路径测试、缺少镜像成功路径结构的边缘情况测试）
- 以 80-90% 实现的功能，而 100% 可以适度添加额外代码实现

#### 时间窗口安全
- 假设"今天"覆盖 24h 的日期键查找 — 早上 8 点 PT 的报告在今天的键下只能看到午夜 → 上午 8 点
- 相关功能之间的时间窗口不匹配 — 一个使用每小时桶，另一个对同一天数据使用每日键

#### 边界类型强制转换
- 值穿越 Ruby→JSON→JS 边界时类型可能改变（数字 vs 字符串） — hash/摘要输入必须在序列化前标准化类型
- Hash/摘要输入在序列化前不调用 `.to_s` 或等效方法 — `{ cores: 8 }` 和 `{ cores: "8" }` 产生不同的哈希

#### 视图/前端
- 在部分中的内联 `<style>` 块（每次渲染重新解析）
- 视图中的 O(n*m) 查找（循环中的 `Array#find` 而不是 `index_by` hash）
- DB 结果上的 Ruby 端 `.select{}` 过滤，可以是 `WHERE` 子句（除非故意避免前导通配符 `LIKE`）

#### 分发与 CI/CD 管道
- CI/CD 工作流更改（`.github/workflows/`）：验证构建工具版本匹配项目要求、artifact 名称/路径正确、密钥使用 `${{ secrets.X }}` 而不是硬编码值
- 新 artifact 类型（CLI 二进制、库、包）：验证发布/释放工作流存在且目标平台正确
- 跨平台构建：验证 CI 矩阵覆盖所有目标 OS/组合，或记录哪些未测试
- 版本标签格式一致性：`v1.2.3` vs `1.2.3` — 必须在 VERSION 文件、git 标签和发布脚本之间匹配
- 发布步骤幂等性：重新运行发布工作流不应失败（例如，在 `gh release create` 之前 `gh release delete`）

**不要标记：**
- 具有现有自动部署管道的 Web 服务（Docker 构建 + K8s 部署）
- 不在团队外部分发的内部工具
- 仅测试的 CI 更改（添加测试步骤，不是发布步骤）

## 先修复启发式

这个启发式同时被 `/review` 和 `/ship` 引用。它决定代理是自动修复发现还是询问用户。

```
AUTO-FIX（代理不询问直接修复）：          ASK（需要人工判断）：
├─ 死代码 / 未使用变量                  ├─ 安全（认证、XSS、注入）
├─ N+1 查询（缺少预加载）                ├─ 竞态条件
├─ 矛盾代码的陈旧注释                   ├─ 设计决策
├─ 魔术数字 → 命名常量                  ├─ 大型修复（>20 行）
├─ 缺少 LLM 输出验证                    ├─ 枚举完整性
├─ 版本/路径不匹配                       ├─ 移除功能
├─ 赋值但从不读取的变量                 └─ 任何更改用户可见行为的
└─ 内联样式、O(n*m) 视图查找
```

**经验法则：** 如果修复是机械化的，高级工程师会不讨论就应用它，那么它是 AUTO-FIX。如果对修复合理工程师可能不同意，那么它是 ASK。

**严重发现默认倾向于 ASK**（它们本质上更危险）。
**信息性发现默认倾向于 AUTO-FIX**（它们更机械化）。

## 抑制 — 以下情况不要标记

- "X 与 Y 冗余"当冗余无害且有助于可读性时（例如 `present?` 与 `length > 20` 冗余时无害）
- "添加一个注释解释为什么选择这个阈值/常量" — 阈值在调优期间变化，注释会腐烂
- "这个断言可以更严格"当断言已经涵盖该行为时
- 建议仅为了改变一致性（将值包装在条件语句中以匹配另一个常量的保护方式）
- "正则表达式不处理边缘情况 X"当输入受限且 X 从未在实践中发生时
- "测试同时练习多个守卫" — 这没问题，测试不需要隔离每个守卫
- 评估阈值更改（max_actionable、最低分数）— 这些是经验性调优的，会不断变化
- 无害的 no-op（例如对永远不会在数组中的元素 `.reject`）
- 任何你已经在你审查的 diff 中处理过的内容 — 在评论之前阅读完整 diff
