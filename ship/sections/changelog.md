## 步骤 13：变更日志（自动生成）

1. 读取 `CHANGELOG.md` 头部以了解格式。

2. **首先，列出分支上的每个提交：**
   ```bash
   git log <base>..HEAD --oneline
   ```
   复制完整列表。计提交数。你将以此作为清单。

3. **读取完整 diff** 以了解每个提交的变更：
   ```bash
   git diff <base>...HEAD
   ```

4. **按主题分组提交** 再开始撰写。常见主题：
   - 新功能 / 能力
   - 性能改进
   - 错误修复
   - 死代码移除 / 清理
   - 基础设施 / 工具 / 测试
   - 重构

5. **撰写变更日志条目** 涵盖所有分组：
   - 如果分支上现有的变更日志已涵盖某些提交，则将其替换为新版本的统一条目
   - 将变更分类到合适的小节：
     - `### Added` — 新功能
     - `### Changed` — 现有功能的变更
     - `### Fixed` — 错误修复
     - `### Removed` — 已移除的功能
   - 撰写简洁、描述性的项目符号
   - 插入到文件头部之后（第 5 行），使用今天的日期
   - 格式：`## [X.Y.Z.W] - YYYY-MM-DD`
   - **语气：** 以用户现在能**做什么**之前的做不到的事开篇。使用简单语言，而非实现细节。绝不提及待办事项.md、内部跟踪或面向贡献者的细节。

6. **交叉核对：** 将变更日志条目与步骤 2 的提交列表对比。
   每个提交必须映射到至少一个项目。若有任何提交未被覆盖，
   立即补上。如果分支有 N 个提交涉及 K 个主题，则变更日志必须
   反映所有 K 个主题。

**请勿要求用户描述变更。** 从 diff 和提交历史中推断。
